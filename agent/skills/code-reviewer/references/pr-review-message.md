# PR review message template

Nexo Co-Review turns your JSON `findings` into GitHub comments. **Do not** post comments via MCP or invent your own markdown layout — follow this template so every review looks the same.

## How findings become comments

| Condition | Where it appears | Rendered format |
|-----------|------------------|-----------------|
| `file_path` + `line_start` set, line is in the PR diff | Inline on the diff | `**[SEVERITY] title**` + body |
| Same, but line not in diff | Summary comment only | `### [SEVERITY] title (\`path\`:line)` + body |
| No `file_path` / `line_start` | Summary comment only | `### [SEVERITY] title` + body |
| Empty `findings` array | Summary comment | `## Nexo Co-Review — org/repo#N` + `No issues found. LGTM!` + attribution footer |

Severity in rendered comments is always **uppercase** (`CRITICAL`, `WARNING`, `INFO`, `SUGGESTION`).

## Attribution footer (required on summary comment)

Every **PR summary comment** must end with this footer (Nexo adds it automatically when posting — do not put it in JSON `body` fields):

```markdown
---

*Review generated by [Nexo Co-Review](https://github.com/Nexo-Agent/code-review).*
```

- Always credit **Nexo Co-Review** as the review source
- Link must point to the project repository: `https://github.com/Nexo-Agent/code-review`
- Inline diff comments do **not** repeat the footer (summary comment carries attribution once per review)

## `title` rules

- One line, **≤ 80 characters**, imperative mood (e.g. "Add null check before dereference")
- No trailing period; no severity prefix (do not start with `[WARNING]`)
- Describe the **issue**, not the file name

## `body` template (required)

Every finding `body` **must** use exactly these three Markdown sections, in this order:

```markdown
**Problem:** What is wrong or risky in the changed code (1–3 sentences).

**Impact:** What could break, leak, or degrade if merged as-is.

**Recommendation:** Concrete fix, test to add, or follow-up action (code snippet optional).
```

Additional rules:

- Do **not** repeat severity or title inside `body`
- Do **not** add extra top-level headings (`##`, `###`) in `body`
- Keep each section to 1–4 sentences; use bullet lists only under **Recommendation**
- For `suggestion` severity, **Impact** may be one short line (e.g. "Optional — improves readability.")
- For `critical` / `warning`, **Impact** must state user-facing or production risk

## Location fields

- `file_path`: repo-relative path (e.g. `backend/app/api/v1/webhooks.py`)
- `line_start`: first line of the issue in the **PR diff** (prefer added/changed lines)
- `line_end`: optional; use when the issue spans multiple lines

Set `file_path` + `line_start` whenever the issue maps to a specific line in the diff. Omit both for PR-wide or architectural notes.

## Example finding

```json
{
  "severity": "warning",
  "title": "Validate webhook signature before parsing body",
  "body": "**Problem:** The handler parses JSON before verifying the GitHub HMAC signature, so malformed payloads are processed unnecessarily.\n\n**Impact:** Wastes work on invalid requests and makes error paths harder to reason about; a logic bug in parsing could mask auth failures.\n\n**Recommendation:** Call signature verification first, return 401 on mismatch, then parse the body. Add a test that sends an invalid signature and expects 401.",
  "file_path": "backend/app/api/v1/webhooks.py",
  "line_start": 42,
  "line_end": 48
}
```

## Example rendered summary comment

```markdown
## Nexo Co-Review — org/repo#42

### [WARNING] Validate webhook signature before parsing body (`backend/app/api/v1/webhooks.py`:42)
**Problem:** The handler parses JSON before verifying the GitHub HMAC signature, so malformed payloads are processed unnecessarily.

**Impact:** Wastes work on invalid requests and makes error paths harder to reason about; a logic bug in parsing could mask auth failures.

**Recommendation:** Call signature verification first, return 401 on mismatch, then parse the body. Add a test that sends an invalid signature and expects 401.

---

*Review generated by [Nexo Co-Review](https://github.com/Nexo-Agent/code-review).*
```

## Example rendered inline comment

```markdown
**[WARNING] Validate webhook signature before parsing body**

**Problem:** The handler parses JSON before verifying the GitHub HMAC signature, so malformed payloads are processed unnecessarily.

**Impact:** Wastes work on invalid requests and makes error paths harder to reason about; a logic bug in parsing could mask auth failures.

**Recommendation:** Call signature verification first, return 401 on mismatch, then parse the body. Add a test that sends an invalid signature and expects 401.
```

## Clean review (LGTM)

When there are no actionable issues, return:

```json
{ "findings": [] }
```

Do not add a synthetic "LGTM" finding — the platform posts the summary automatically.
